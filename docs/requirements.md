## 背景・目的

個人事業主・小規模法人向けに、領収書画像・PDFのOCR抽出、勘定科目自動判定、ルールベース補正、スプレッドシート保存、監査ログ管理を自動化する。銀行データとの照合およびAI駆動レビューにより精度と運用効率を向上させることを目的とする。

## 対象ユーザー・ロール

## ユーザーストーリー一覧

### US-01 単一領収書アップロード
ユーザーが領収書1件をカスタムGPTに送信すると、システムはGoogle Driveの指定フォルダ（`DRIVE_FOLDER_ID`）に一時ファイルとして保存し、`receiptKey`（Drive fileId）を返却する。エラー時は `STAGE_DOWNLOAD_FAILED` を返す。アップロード中は一切の質問を行わず、完了メッセージのみを返す。

### US-02 複数領収書アップロード（最大10件）
US-01と同様の処理を最大10件まで並列処理する。一部失敗時は `partial_ok` ステータスで成功分のみ返却する。

### US-03 領収書なし経費登録
家賃・交通費など領収書が存在しない経費を直接登録できる。`no_receipt=true` フラグを付与し、`reconcileStatus = unreconciled` で登録する。

### US-04 OCR自動抽出
アップロードされた領収書からOpenAI Visionを使用して以下を抽出する。抽出対象は `document_date`、`transaction_date`、`vendor`、`summary`、`debit_account`、`amount`、`currency`、`taxCategory`、`qualifiedInvoiceNumber`、`confidence` である。抽出失敗時は `EXTRACT_FAILED` を返す。

### US-05 勘定科目自動判定
信頼度（`confidence`）が0.8以上の場合は自動確定し、0.8未満の場合は `needsReview=true` を設定する。判定結果は変更履歴に記録する。

### US-06 飲食費科目判定フロー
飲食系領収書（会議費・交際費候補）はデフォルトで `needsReview=true` とする。月末レビュー時にAIが `memo`（参加者・目的）を確認し、会議費または交際費に確定する。`memo` が不明・曖昧な場合は引き続き `needsReview=true` を維持する。

### US-07 expenseTypeフラグ管理
`expenseType` のデフォルトは `corporate` とする。`personal` への変更はシート直接編集で行い、`onEdit` トリガーが変更履歴に記録する。`corporate` の貸方科目は短期借入金（銀行照合で法人口座確認後は普通預金に更新）、`personal` の貸方科目は短期借入金とする。

### US-08 家事按分15%ルール
`debit_account` が地代家賃・水道光熱費・通信費のいずれかで、かつ `vendor` が `HOME_OFFICE_WHITELIST`（ソフトバンク、NTTファイナンス、東京ガス、東京都水道局等）に一致する場合、`applyHomeOfficeRule=true` を設定し、`bookedAmount = round(originalAmount × 0.15)` を計算する。銀行照合キーは `originalAmount` を使用する。

### US-09 外貨領収書処理
外貨領収書を検知した場合、取引日のレートを77銀行仲値ページからOpenAI経由で取得し、仮円換算額を算出する。URLは `buildFxRateUrl_(currency, date)` 関数で動的生成する（例: `https://www.77bank.co.jp/kawase/usd2026.html`）。取得失敗時は `needsReview=true`、`bookedAmount=null` とする。対応通貨はUSDを基本とし、77銀行がページを提供していない通貨は `needsReview=true` でフォールバックする。為替レート確定後はMoneyForward照合により `bookedAmount` を更新する。

### US-10 年額サブスク月割り処理（P0）
年額・長期契約を検知した場合、前払費用として計上し月割り行を自動生成する。月割り額は `round(originalAmount / 月数)` とし、最終月は端数調整する。解約時は残額を一括費用計上する。月割り行は定期経費マスターと連携して自動管理する。短期前払費用の特例適用可否は顧問税理士に確認すること。

### US-11 適格請求書番号正規化
全角→半角変換、ハイフン除去、大文字化を行い、`^T[0-9]{13}$` パターンで検証する。不正な場合は `needsReview=true` を設定する。

### US-12 3種類の日付管理
`document_date`（発行日）、`transaction_date`（取引日・検針日）、`payment_date`（支払日・銀行照合で確定）の3フィールドを独立管理する。使用期間は `service_period_start` / `service_period_end` で管理し、`transaction_date` は常に単一日付型を維持する。

### US-13 計上日基準
計上日の優先順位は `payment_date → transaction_date → document_date` とする。Script Properties の `BOOKING_DATE_BASIS = "payment_date"` で管理する。

### US-14 needsReview自動設定
以下の条件のいずれかに該当する場合、`needsReview=true` を自動設定する。対象条件は `confidence < 0.8`、日付欠損、vendor欠損、金額欠損、外貨かつ円金額なし、FXレート取得失敗、年額契約検出、適格番号不正、家事按分対象か不明、科目不確定、飲食メモ判定困難、重複疑い、グローバルレビューアラートである。

### US-15 AIレビューモード（B+Cハイブリッド）
月末または手動トリガーでレビューモードに入る。AIは `needsReview=true` の行とグローバルアラートを集約し、修正提案を一括提示する（モードB）。複雑なケースはAIが対話的に質問して確定する（モードC）。ユーザーが承認後にシートを更新する。アップロード中は一切のレビュー通知を行わない。

### US-16 月末通知設定
通知タイミングはScript Propertiesで設定する。`REVIEW_TRIGGER`（`manual` / `month_end`）、`TRANSPORT_TIMING`（`manual` / `month_end`）のデフォルトはいずれも `month_end` とする。

### US-17 ファイル命名規則
ファイル名は `YYYYMMDD_勘定科目_取引先_金額_識別子.ext` とする。日付の優先順位は `document_date → transaction_date → payment_date → 登録日時` とする。金額は `originalAmount`（外貨はそのまま）を使用する。識別子はランダム6文字英数字を必ず付与する。禁止文字（`/ \ : * ? " < > |` およびスペース）はアンダースコアに置換し、連続アンダースコアは1つに圧縮する。全体が100文字を超える場合はvendorを先頭20文字で切り詰める。全失敗時は `receipt_{登録日時}.ext` とする。保存先は `DRIVE_FOLDER_ID = 1ZsXakvuc-XO7N3V3lmtNEEeKnsijLM2C` のフラット構造とする。

### US-18 銀行照合（reconcile）
照合ソースはGMOあおぞらネット銀行（CSV/PDF）、MoneyForward（CSV/PDF/画像）、東山口信用金庫・ゆうちょ銀行（通帳写真）とする。照合キーは `originalAmount` + 取引先とし、日付許容範囲は±1日とする。照合成功時は `payment_date` を確定し、`mfTransactionId` を記録し、貸方科目を普通預金（該当口座）に更新し、`reconcileStatus = reconciled` に変更する。

### US-19 reconcileStatusの5段階管理

| ステータス | 意味 | 貸方科目 |
|:--|:--|:--|
| `unreconciled` | 未照合 | 短期借入金 |
| `reconciled` | 照合確定済み | 普通預金（照合口座）or 短期借入金 |
| `reconciled_modified` | 照合後に手動修正あり | 同上 |
| `cash_payment` | 現金払い・照合不要と確定 | 短期借入金 |
| `reconcile_pending` | 照合待ち（通帳待ち等） | 短期借入金 |

現金払い・個人カード払いを問わず貸方科目は短期借入金で統一する。`unreconciled` が30日以上経過した行は月末レビュー時にAIが確認する。

### US-20 照合パターン自動適用
照合確認済みのパターン（金額±5%、日付±1日、取引先一致）を学習し、次回以降自動適用する。外貨照合は日付±1日・金額完全一致とし、複数候補がある場合はAIが全候補提示してユーザーが選択する。

### US-21 MFホワイトリストによる漏れ検知
MFデータ取り込み時にホワイトリストシート（`MFホワイトリスト`）でフィルタリングし、ヒットした取引のみを照合対象とする。マッチしなかった取引は完全無視する。照合失敗（経費シートに対応行なし）の取引は全体レビューアラートシートに `alertType = "unmatched_bank_transaction"` として記録し、月末レビュー時に提示する。ユーザーが「不要」と回答した場合は学習ログに記録して次回以降スキップする。

### US-22 月末抜け漏れ検知
月末トリガーで定期経費マスターの登録予定項目と実際の登録状況を照合し、未登録項目をAIが提案する（例：「今月の東京ガスの登録がありません」）。

### US-23 定期経費自動入力
定期経費マスターシートに登録された経費を月末23:00のタイムトリガーで自動入力する。マスターには有効開始日・有効終了日を持たせ、金額変更時は新レコードを追加することで過去行への遡及を防ぐ。自動入力失敗時はエラーをアラートシートに記録しAIが翌日通知する。

### US-24 交通費自動計算
ユーザーが経路を入力すると、OpenAI経由で取引日時点のIC運賃を取得する。運賃は有効期間付きキャッシュシートで管理し、同一期間の2レコードが同額の場合はキャッシュを使用、異なる場合は期間判定で適切なレートを適用する。取得失敗時は `needsReview=true` とする。

### US-25 交通費マスター管理
よく使う経路をマスターシートに登録し、経路名で選択できる。マスターには運賃を持たせず、経路情報のみを保持する。運賃は取引日を指定してOpenAIに問い合わせる。

### US-26 論理削除
経費シートに `isDeleted`、`deletedAt`、`deletedBy`、`deleteReason` 列を追加する。物理削除は行わない。論理削除された行は集計・照合・グローバルレビューの対象外とするが、監査ログ・変更履歴には全件記録する。誤削除時は `isDeleted=false` に戻すことで復元する。論理削除されたDriveファイルは `DRIVE_DELETED_FOLDER_ID` フォルダに移動する。

### US-27 月割り行の連動論理削除
親行（年額サブスク）を論理削除する際、AIが月割り子行の状況を分析し確認メッセージを提示する。計上済み行（`payment_date` 確定済み）は残すことを推奨し、未来月の未計上行は削除を推奨し、当月行は個別確認を行う。

### US-28 finalize重複呼び出し制御
同一 `receiptKey` に対して `finalize` を2回呼び出した場合、`ALREADY_FINALIZED` エラーを返す。再処理が必要な場合は `"refinalize": true` フラグを明示的に付与して呼び出す。

### US-29 シート書き込み一元化
全てのシート書き込みは `appendRow_()` または `updateRow_()` 専用関数を必ず経由する。`sheet.getRange().setValue()` の直接呼び出しを禁止し、専用関数内で `logChange_()` を呼び出すことで、APIからの更新・月末自動入力・銀行照合による更新を全て変更履歴に記録する。

### US-30 全体レビューによる横断的異常検知
レイヤー1としてGASルールベースチェックを実行する。チェック対象は重複エントリ検出（同日・同金額・同取引先の複数行）、月次異常検知（科目別金額が前月比 `ANOMALY_THRESHOLD_RATE` 倍超かつ `ANOMALY_MIN_AMOUNT` 円以上）、為替レート整合性（同日同通貨で異なるレート）、家事按分適用漏れ、前払費用残高整合性である。レイヤー2としてOpenAI AIによる自由観察を実行し、定義外の異常を検知する。結果は全体レビューアラートシートに記録する。ユーザーが「不要」とフィードバックした場合は学習ログにルールを記録し次回以降スキップする。ルール適用ログに毎回の適用・スキップ結果を記録する。

### US-31 フィードバック学習閾値管理
ユーザーフィードバックで学習されたルールは学習ログシートに記録し、次回以降の判定に自動適用する。ルール優先順位はユーザー承認済みルール＞AI自動学習ルール＞Script Properties閾値＞AIデフォルト判断とする。競合時はより具体的なルールを優先する。誤学習時はカスタムGPT上で「このルールを削除して」と指示することで `status=revoked` に更新する（論理削除）。未使用ルールは `LEARNING_RULE_EXPIRE_MONTHS = 12` ヶ月で自動失効し、`LEARNING_RULE_MAX = 50` 件を上限とする。

## スコープ外

会計ソフトへの直接連携、外貨APIの直接呼び出し、オンライン適格請求書番号照会、残高自動取得、非同期リトライ、スキャンPDFの品質保証、マルチユーザー対応は対象外とする。

---

## 非機能要件

パフォーマンスとして `finalize` は最大10件、GAS6分制限を遵守する。信頼性として `partial_ok` ハンドリングを実装し、月末タイムトリガーは自動再設定する。セキュリティとしてAPIトークン認証を必須とし、Driveファイルは非公開とし、トークン漏洩時はScript Propertiesの `API_TOKEN` を即時書き換えることで無効化する。保守性としてScript Propertiesで全設定を管理し、ハードコーディングを禁止する。拡張性として銀行・通貨・15%ホワイトリスト・経路の追加をシートまたはScript Propertiesで対応可能とする。
